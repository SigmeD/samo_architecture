# ADR-002: Родительский доступ — отдельный аккаунт

**Статус:** Принято (27.03.2026)
**Источник:** `docs/etap-3-dnm-plan.md` §1

## Контекст
В ТС 5.9 (строка 319) указано: «Родительский кабинет реализуется как раздел в кабинете ученика, а не отдельный аккаунт». Заказчик изменил решение.

## Решение
Родитель — отдельный аккаунт (`User` с ролью `cr5-roditel-dnm`) со связью через `DnmParentLink → DnmEnrollment`.

## Обоснование
- Один родитель может иметь нескольких детей в разных филиалах.
- Родитель имеет собственные финансовые операции (оплаты, история, акты).
- Разделение аккаунтов проще для безопасности — ребёнок не видит финансовые данные.
- Один логин с ребёнком не используется.

## Последствия
- (+) Чистая ролевая модель, совместимая с RBAC системы.
- (+) Родитель видит прогресс нескольких детей в одном кабинете.
- (−) Нужен отдельный flow регистрации «Привязка родителя к ученику».
- (!) Требовалось обновление ТС 5.9 / ФО 3.9 (бизнес-правило 1).

## Как реализовано (факт)
`DnmParentLink` со статусами `PENDING_PAYMENT / ACTIVE / FROZEN / REJECTED`; онбординг через `DnmParentChildInvite` и приглашения. См. `docs/STATUS.md`.
