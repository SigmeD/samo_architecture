## Назначение

Кадровый контур школы/сети. HR (`op-hr-dnm`) отвечает за полный жизненный цикл персонала: от поиска кандидата до удержания нанятого — и является **владельцем KPI-каркаса персонала и базы данных сотрудников** (system of record).

Ядро — **воронка найма → онбординг-Академия → аттестация → допуск → удержание**. HR ведёт ПЕРСОНАЛ (не учеников, не учебные данные), проектирует KPI-карты ролей (задаёт ЧТО мерить), хранит кадровые данные (ставка = кадровый атрибут, не платёж) и передаёт их бухгалтеру + движку SPEC-KPI-PAYOUT для расчёта выплат.

**Подчинение (OQ-ORG-02 — на подтверждении):** директор школы (Модель А) / уровень сети (Модель Б).

**Границы (scope):** HR **НЕ видит учебных данных** (≠ куратор/ст.куратор) и **НЕ видит финотчётов/P&L/роялти** (≠ бухгалтер/владелец). HR **НЕ считает выплат** — это движок SPEC-KPI-PAYOUT + бухгалтер (граница OQ-HR-02): HR задаёт ЧТО мерить (KPI-карты ролей, метрики, веса, пороги, таргеты), а движок считает выплату (Фикс+Бонус−Штраф). HR **НЕ распределяет учеников** (это администратор — HR распределяет ПЕРСОНАЛ). Ставка в базе персонала = **кадровый атрибут** (не платёж, не платёжная операция): питает ФОТ через бухгалтера, но HR её не выплачивает.

## Ключевые инварианты (не нарушать)

- **HR не считает выплат** (граница OQ-HR-02): KPI-карты ролей — в HR; расчёт Фикс+Бонус−Штраф — движок SPEC-KPI-PAYOUT + бухгалтер.
- **Ставка = кадровый атрибут**, не платёж и не платёжная операция; расчёт выплаты — бухгалтер.
- **HR не видит учебных данных** (занятия/ДЗ/оценки/солары ≠ кадровый контур).
- **HR не видит финотчётов/P&L/роялти** — конфиденциально (владелец+бухгалтер).
- **HR не распределяет учеников** — только персонал; распределение учеников = администратор.
- **ПДн персонала — OQ-HR-03** (на подтверждении); хранение/обработка персональных данных сотрудников.
- **Обезличено** (только роли); без доли роялти.

## Ядро: Воронка найма → онбординг → аттестация → допуск → удержание

Кадровый контур HR (REG-DNM-HR-001 v1.1 §2): от источника кандидата до удержания нанятого. Контур онбординга находится **внутри HR** (решение владельца 12.06, OQ-HR-01 закрыт); подчинение HR — директору школы (Модель А) / сети (Модель Б), OQ-ORG-02 на подтверждении.

| # | Шаг | Суть | Источник |
|---|---|---|---|
| 1 | 🔎 **Источник → скрининг** | Поиск и привлечение кандидатов по ролям (МОП/РОП/куратор/ст.куратор/админ/маркетолог/таргетолог): шаблоны вакансий, обзвон, первичный скрининг анкет, наполнение кадрового резерва. Метрики воронки: обзвонил → пригласил → дошёл до теста. | REG-DNM-HR-001 v1.1 §2.1 |
| 2 | 📝 **Тест при приёме** | Кандидат проходит тест из банка тестов по роли → балл → допуск к собеседованию. HR ведёт банк тестов по ролям, пороги допуска. | REG-DNM-HR-001 v1.1 §2.2 |
| 3 | 🗣️ **Собеседование → оффер** | Интервью по результатам теста, оценка соответствия роли, формирование и выдача оффера принятому кандидату. | REG-DNM-HR-001 v1.1 §2.1/§2.2 |

| 5 | ✅ **Аттестация → допуск** | Баллы по результатам онбординга = аттестация → пробный период → допуск к работе с клиентами. Результаты тестов/аттестаций пишутся в базу персонала. | REG-DNM-HR-001 v1.1 §2.2 |
| 6 | 📉 **Удержание** | Мониторинг текучести по ролям/школам, exit-интервью при выходе, программы удержания — антитекучка как замыкание контура. | REG-DNM-HR-001 v1.1 §2.6/§3 |

## Домены и функции (8)

### 🎯 Воронка рекрутинга
Стадии и метрики воронки: обзвонил → пригласил → дошёл до теста → нанято (конверсия по этапам) · шаблоны вакансий по ролям; первичный скрининг анкет · кадровый резерв по ролям: МОП / РОП / куратор / ст.куратор / админ / маркетолог / таргетолог · ⏸ ATS-автоматизация воронки — planned (conditional, ADR-0002): пока ведётся вручную.
_Источник: REG-DNM-HR-001 v1.1 §2.1; ADR-0002._

### 📝 Банк тестов при приёме
HR загружает и ведёт тесты при приёме по ролям (банк тестов) · кандидат проходит тест → балл → допуск к собеседованию / пробному · пороги допуска по ролям; привязка теста к вакансии.
_Источник: REG-DNM-HR-001 v1.1 §2.2._

### 🎓 Академия онбординга / аттестация
Role-based обучение: видео+текст обязанностей по модулю роли → тест после модуля · баллы = аттестация → пробный период → допуск к работе с клиентами · платформа = M6-LMS; контур онбординга — внутри HR (решение владельца 12.06, OQ-HR-01 закрыт) · результаты тестов/аттестаций пишутся в базу данных персонала.
_Источник: REG-DNM-HR-001 v1.1 §2.2; M6-LMS._

### 📊 KPI-карты персонала
HR проектирует и ведёт KPI-карты ролей: метрики, веса, пороги, таргеты — задаёт ЧТО мерить · граница (OQ-HR-02): HR задаёт что мерить + таргеты; движок SPEC-KPI-PAYOUT считает ВЫПЛАТУ (Фикс+Бонус−Штраф). HR не считает выплат · KPI-карты питают репутационные рейтинги ролей (SPEC-DNM-RATING §4) · согласование KPI-карт ролей и оргструктуры — с директором школы.
_Источник: REG-DNM-HR-001 v1.1 §2.3; SPEC-KPI-PAYOUT-001 v1.1; SPEC-DNM-RATING-001 v1.2 §4._

### 🗂️ База данных персонала
System of record по сотрудникам: карточка (роль, школа, дата приёма, статус, документы, история ролей/переводов, результаты тестов/аттестаций, табель/отпуска) · ставка хранится как КАДРОВЫЙ АТРИБУТ — не платёж и не платёжная операция (выплату считает бухгалтер) · питает ФОТ (через SPEC-KPI-PAYOUT §2.4 + бухгалтер), оргструктуру и IAM/M1 (создание/деактивация учёток) · ПДн персонала — OQ-HR-03 (на подтверждении).
_Источник: REG-DNM-HR-001 v1.1 §2.4; SPEC-KPI-PAYOUT-001 v1.1 §2.4._

### 🤝 Внутренняя политика / дисциплина / медиация
Регламенты входа/выхода сотрудника, корпоративный этикет, контроль соблюдения · медиация конфликтов между сотрудниками; дисциплинарные процедуры · опора на регламент этики (REG-ETHICS-001).
_Источник: REG-DNM-HR-001 v1.1 §2.5; REG-ETHICS-001._

### 📉 Удержание (антитекучка)
Мониторинг текучести персонала по ролям и школам · exit-интервью при выходе сотрудника; разбор причин · программы удержания и снижения текучести.
_Источник: REG-DNM-HR-001 v1.1 §2.6._

### 📈 KPI самого HR
Воронка-конверсия найма; time-to-hire (скорость закрытия вакансии) · качество найма: retention 3 / 6 / 12 мес; текучесть ↓; укомплектованность штата · таргеты KPI самого HR — OQ-HR-04 (на подтверждении).
_Источник: REG-DNM-HR-001 v1.1 §3._

## Модули (бизнес-процессы, 7)

1. **Воронка рекрутинга** — стадии и метрики воронки (обзвонил → пригласил → дошёл до теста → нанято), шаблоны вакансий, кадровый резерв по ролям. ⏸ ATS-автоматизация — planned (conditional, ADR-0002). REG-DNM-HR-001 v1.1 §2.1.
2. **Банк тестов при приёме** — HR ведёт тесты при приёме по ролям; кандидат проходит → балл → допуск к собеседованию/пробному; пороги допуска по ролям. REG-DNM-HR-001 v1.1 §2.2.
3. **Академия онбординга / аттестация** — role-based онбординг на платформе M6-LMS (видео+текст → тест после модуля → баллы = аттестация → пробный → допуск). Контур внутри HR (OQ-HR-01 закрыт). REG-DNM-HR-001 v1.1 §2.2.
4. **KPI-карты персонала** — HR проектирует KPI-карты ролей (метрики, веса, пороги, таргеты) — задаёт ЧТО мерить; движок SPEC-KPI-PAYOUT считает выплату (граница OQ-HR-02); KPI питают рейтинги (SPEC-DNM-RATING §4). REG-DNM-HR-001 v1.1 §2.3.
5. **База данных персонала** — system of record по сотрудникам (роль, школа, статус, документы, история, тесты/аттестации, табель). Ставка = кадровый атрибут (не платёж); питает ФОТ/оргструктуру/IAM. ПДн — OQ-HR-03. REG-DNM-HR-001 v1.1 §2.4.
6. **Внутренняя политика / дисциплина / медиация** — регламенты входа/выхода, этикет, контроль; медиация конфликтов, дисциплинарные процедуры (REG-ETHICS-001). REG-DNM-HR-001 v1.1 §2.5.
7. **Удержание (антитекучка)** — мониторинг текучести по ролям/школам, exit-интервью, программы удержания. REG-DNM-HR-001 v1.1 §2.6.

> Все модули в статусе `planned` (продукт роли в коде не реализован — карта намерения).

## Открытые вопросы (показаны в атласе как «на подтверждении»)

- **OQ-ORG-02:** подчинение HR — директор школы (Модель А) или уровень сети (Модель Б)? На подтверждении владельца.
- **OQ-HR-02:** граница «HR задаёт KPI-карты / движок+бухгалтер считает выплату» — точный интерфейс передачи данных и RBAC на редактирование KPI-карт ролей.
- **OQ-HR-03:** хранение и обработка ПДн персонала — правовой режим, доступы, срок хранения.
- **OQ-HR-04:** таргеты KPI самого HR (конверсия найма, time-to-hire, retention) — нормативные пороги не зафиксированы.
- **ATS ⏸ planned (conditional ADR-0002):** автоматизация воронки рекрутинга — зависит от ратификации ADR-0002 (централизация платёжных/операционных потоков).
