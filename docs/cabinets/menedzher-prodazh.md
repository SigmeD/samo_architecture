## Назначение

Кабинет менеджера по продажам ведёт **воронку лида-родителя и доводит её до договора**: приём лидов от маркетолога, квалификация в CRM (анкета → рекомендация продукта + скрипты), выдача демо-доступа, онбординг и пробное, постоянный чат с родителем-лидом, омниканальный дожим (WhatsApp ↔ CRM), оформление договора на оплату.

**Реверс (решение владельца 09.06):** прежняя поправка к постеру («договор/оплату ведёт администратор, менеджер только до передачи; пробное проводит куратор») **ОТОЗВАНА** — намерение совпало с каноном `CONV-RBAC-DNM-001 v1.4 §6` и исходным постером 02.06. Новая логика:

- **Договор:** менеджер **ОФОРМЛЯЕТ** договор на оплату с лидом → **АДМИНИСТРАТОР ПОДТВЕРЖДАЕТ** → оплата проходит через **БУХГАЛТЕРИЮ** (бухгалтер подтверждает факт оплаты). Рассрочку/дебиторку/возвраты/расторжение ведут финансы/администратор.
- **Пробное:** менеджер проводит онбординг и пробное **САМ ИЛИ назначает куратора** из расписания; пробное менеджера идёт под **видеофиксацией/контролем качества старшего куратора** (close-rate считается, единый стандарт).

**Границы (scope):** менеджер не видит общий раздел «Сделки» (только свою воронку); не редактирует продукты и программу; scope — только лиды-родители ДНМ. Продукты юрлицам/инвесторам — отдельная CRM Samo Global, вне этого кабинета. Speed-to-lead (первый контакт в минуты) — норматив и метрика. Обезличено (только роли).

## Ключевые инварианты (не нарушать)

- Менеджер ведёт **воронку лида-родителя и ОФОРМЛЯЕТ договор на оплату** — доводит лид до договора.
- **Договор оформляет менеджер → администратор ПОДТВЕРЖДАЕТ → оплата через бухгалтерию** (бухгалтер подтверждает факт оплаты). Совпадает с `CONV-RBAC-DNM-001 v1.4 §6` (Мен: приём оплаты/рассрочка CRU) — расхождение СНЯТО.
- **Рассрочку, дебиторку, возвраты, расторжение** ведут администратор/финансы; зачисление — у администратора (мастер).
- **Пробное проводит менеджер САМ ИЛИ назначенный куратор** из расписания; пробное менеджера — под видеофиксацией/контролем качества старшего куратора (close-rate считается, единый стандарт).
- **Постоянный чат менеджера с родителем-ЛИДОМ** на всём пути ДО подписания договора (родитель как лид, коммуникация постоянная).
- Омниканальный дожим WhatsApp Business API ↔ CRM: вся переписка в карточке (аудит); история не теряется при уходе менеджера.
- **Speed-to-lead** (первый контакт в минуты) — норматив и метрика.
- Scope — только лиды-родители ДНМ; **B2B/инвесторы — отдельная CRM Samo Global**, вне кабинета.
- Обезличено (только роли).

## Домены и функции (10)

### 📊 Мои KPI · _только просмотр_
Воронка: лиды → демо → пробные → договор → подтверждено, с конверсией по этапам · **конверсия лида «X из 100»** (напр. 4 из 100 / 1 из 100) — метрика успешности воронки (мой KPI) · speed-to-lead — среднее время первого контакта; норматив до 40 мин · период созревания лида в клиента: когорты 3/7/12/30 дней · степень закрытия сделки (close rate) — конверсия «каждый N-й покупает»; пробное менеджера под контролем качества ст.куратора · активность (касания/звонки) — для коучинга, не наказания; цикл лид → договор · конверсия в оплату — по подтверждению администратора и факту оплаты от бухгалтерии.
_Источник: CONV-RBAC-DNM-001 v1.4 §7 (KPI R(св)); проектный док §13; встреча 09.06 §5._

### 📥 Лиды и приоритеты
Дневной список лидов по скорингу; «горячие» — первыми · источник лида (таргет/органика/реклама/рекомендация) — фиксируется · каденс касаний: 6–8 за первые 48 ч (звонки + сообщения) · гигиена воронки: флаги «протухших», авто-переназначение без реакции.
_Источник: проектный док §8/§16; REG-DNM-MARKETING-001 v1.1._

### 👤 Карточка клиента (CRM)
Клиент = родитель (физлицо): контакты + мессенджеры (WhatsApp/Telegram); менеджер добавляет, не меняет основной · входящий запрос вверху → анкета → рекомендация продукта → скрипты · отправка материалов одной кнопкой (афиша + презентация) · дубли: повторная заявка из «Потерянных» → карточка в «Новые», новая строка истории.
_Источник: SPEC-CRM-TZ-001 v1.1; проектный док §6. Без блока юрлица/B2B — это CRM Samo Global._

### 🎬 Демо-доступ и пробные уроки
Выдача демо-доступа родителю+ребёнку (ознакомительные уроки) · онбординг и пробное менеджер проводит **САМ ИЛИ назначает куратора** из расписания · пробное менеджера — под видеофиксацией/контролем качества ст.куратора (единый стандарт, close-rate считается) · коэф. успешности пробного (ориентир 70–80%); close-rate считается независимо от того, кто провёл · после пробного менеджер дожимает (омниканально) · рекомендация совместного урока родитель+ребёнок — менеджер рекомендует, визуализируется в кабинетах родителя и ученика.
_Источник: проектный док §17.1; встреча 04.06; SPEC-M3-DNM-001 v2.7; спека 2026-06-09 реверс._

### 📦 Карточка продукта · _только просмотр_
Описание = скрипт презентации; афиша, расписание · цены в USD (раннее + стандарт × ВИП / Стандарт / Онлайн); конверсия по курсу ЦБ · скрипты презентации и возражений по продукту · менеджер не редактирует — только просмотр в рамках своего ранга.
_Источник: проектный док §7; SPEC-CRM-TZ-001 v1.1._

### 🎬 Топ-20 постов · _только просмотр_
Живые reels с максимальными просмотрами/охватами из соцсетей — готовый продающий контент для дожима · топ-20 роликов под рукой: менеджер отправляет лиду релевантный пост в чат при работе с возражением · связка с «Топ-5 роликов» маркетолога — расширено до топ-20 для продаж (что реально цепляет аудиторию) · метрики поста (просмотры/охват/вовлечённость) видны для выбора — продаём тем, что уже сработало.
_Источник: проектный док §3; REG-DNM-MARKETING-001 v1.1; спека 2026-06-09 реверс (связка с маркетологом)._

### 💬 Омниканальный дожим (WhatsApp ↔ CRM)
WhatsApp Business API интегрирован с CRM: вся переписка попадает в карточку (аудит) · единое окно: менеджер отвечает из CRM или WhatsApp — синхронно; история не теряется при уходе менеджера · шаблоны + журнал отправок; opt-in, вне 24-ч окна — только шаблоны; оферта дублируется почта + WhatsApp · WhatsApp — для лида во входящей воронке; платформенный чат — для уже на платформе.
_Источник: проектный док §9; ARCH-SERVICEMAP-001 v1.1._

### 🔁 Проигранные и прогрев
Закрытие «Проиграна» → обязательная причина из списка + комментарий · клиент → база прогрева; повторный обзвон не раньше 3 мес (маркетинг работает рассылками) · агрегация причин отказа → обратная связь маркетологу (какие источники конвертят).
_Источник: проектный док §12/§16._

### 📈 Статус по моим клиентам
Менеджер **ОФОРМЛЯЕТ** договор на оплату с лидом → **АДМИНИСТРАТОР ПОДТВЕРЖДАЕТ** → оплата через бухгалтерию · статус подтверждения договора и факта оплаты по своим клиентам — видит по своей воронке · бухгалтер подтверждает факт оплаты; рассрочку/дебиторку/возвраты/расторжение ведёт администратор/финансы · менеджер не видит общий раздел «Сделки» по сети — только свою воронку.
_Источник: CONV-RBAC-DNM-001 v1.4 §1/§6 (приём оплаты/рассрочка CRU у Мен; зачисление CRU у админа); проектный док §10/§19; спека 2026-06-09 реверс._

### 🎓 Академия, задачи, связь
Обучающие созвоны (от куратора франшиз); скрипты презентаций/возражений; обучение по продуктам · задачи и уведомления (горячие лиды, перезвоны, протухшие) · чат с куратором/администратором (CRU); контакты · взаимодействие на уровне менеджера: франчайзи-партнёр (своя школа) + руководитель продаж (РОП) головного офиса; ⚠ куратор франшиз СКРЫТ от франчайзи — данные агрегируются и идут наверх.
_Источник: проектный док §14; CONV-RBAC-DNM-001 v1.4 §5 (чат с куратором/админом CRU); встреча 09.06 §5/§6._

## Цепочка продаж/зачисления

`Маркетолог (лид + UTM-источник) → Менеджер (онбординг + пробное САМ или назначает куратора из расписания; постоянный чат с родителем-лидом) → дожим + ДОГОВОР → Администратор подтверждает → оплата через Бухгалтерию → Старший куратор (берёт в свою группу ИЛИ → Куратор)`

«Сам не возьмёт» = старший куратор (двойная роль), не менеджер.
