# Дизайн T3: Сплит «Администратор школы» → «Директор школы» + новый лёгкий «Администратор»

> **Дата:** 2026-06-12 · **Класс:** намерение (канон-авторитет) · **Источник gap:** `2026-06-12-org-roles-canon-gap.md` (T3).
> **Авторитет:** `CONV-ROLES-DNM-001 v1.6 §2а/§2б`, `CONV-RBAC-DNM-001 v1.6`, `CONV-ROLE-HIERARCHY-001 v1.11`, meeting `docs/client/meetings/2026-06-10-...md` §2–§3.
> **Решения владельца:** D2 (rename school-admin→Директор, не переписывая), D3 (новый лёгкий Админ + опц. Лидоруб), **slug-стратегия B** (Директор = `/director`, Админ переиспользует `/school-admin`).

## 1. Проблема

Канон 10.06 развёл управленца и сервис: *«Директор школы заменил ошибочно стоявшего администратора-как-управляющего; администратор возвращается как обслуживающая/распределяющая роль»* (meeting §2). Текущий `school-admin.ts` (14 доменов: дашборды директора AD1–AD6, управление школами, штатка/оргструктура, KPI, отчётность-вертикаль) — **функционально Директор**. Нужен сплит.

## 2. Архитектура (slug-стратегия B)

```
БЫЛО:   /cabinet/school-admin  = жирный кабинет (op-admin-shkoly-dnm), 14 доменов
СТАЛО:  /cabinet/director      = жирный кабинет → «Директор школы ДНМ» (op-direktor-shkoly-dnm), 14 доменов (контент сохранён)
        /cabinet/school-admin  = НОВЫЙ лёгкий «Администратор школы ДНМ» (op-admin-shkoly-dnm)
```

**Почему B:** большинство `crossLinks→school-admin` (чат с родителем/учеником, «договор админ подтверждает», приём оплаты, выдача товаров) семантически = **операционный админ** → они авто-остаются корректными, указывая на новый лёгкий кабинет. Перецепить на `director` нужно только управленческие/вертикальные.

## 3. Разделение функций

**Принцип:** Директор = **оверсайт + управление** (видит всё «аналитикой поверх», настраивает); Админ = **исполнение** (операционный CRU). Перекрытие операционных доменов намеренно (модель «вход как директор = аналитика поверх, операционка делегирована администратору», FR-M3-094).

| Функция | Директор (`/director`) | Админ (`/school-admin`) | Канон |
|---|---|---|---|
| Дашборды AD1–AD6, аналитика, план/факт | ✅ владелец | — | RBAC §7 |

| Настройка KPI-планов/процентов | ✅ | — | RBAC §7 |
| Штатка/оргструктура (решения) | ✅ | табель-исполнение | RBAC §7 |
| Отчётность-вертикаль школа→Франчайзи | ✅ узел вверх | — | HIERARCHY v1.11 |
| Маркетинг/продажи-тоглы (модель A/B) | ✅ | — | RBAC §9 |
| Зачисление-исполнение (подтверждение договора, привязка) | оверсайт | ✅ CRU(шк) | RBAC §1 |

## 4. Новый кабинет «Администратор школы ДНМ» (proposal)

- **slug** `school-admin` · **role** `{ op-admin-shkoly-dnm, "Администратор школы ДНМ", 🏫 }` · **zone** orange · **implStatus** planned.
- **Ядро:** «Приём и распределение» (Договор от менеджера → Подтверждение → Приём оплаты (бухгалтер подтверждает) → Распределение в группу/к ст.куратору). Без P&L.
- **Домены (лёгкие, ~8):**
  1. 📥 Приём и подтверждение (договор-оферту оформляет менеджер → админ подтверждает, привязка ученик/родитель/период)
  2. 💳 Операционные платежи (приём оплат, дебиторка, просрочка, возвраты при расторжении — P&L закрыт)
  3. 🗓️ Расписание и посещаемость (массовая отметка, переносы, **непосещение: 1-е авто-пуш родителю, 2-е звонок админа** — meeting §3)
  4. 🔀 Распределение (после оплаты → ст.куратор/группа; перевод между группами, приостановление)
  5. 💬 Чат-хаб операционки (родитель, ученик-SENIOR, ст.куратор, бухгалтер, менеджер)
  6. 🛒 Выдача за солары (отметка выдачи физтовара, остатки)
  7. ⏱️ Табель (учёт времени персонала → данные для ФОТ бухгалтеру)
  8. ⚙️ Администратор-Лидоруб (опц., dual-mode: фильтр лидов, назначение пробных, распределение по менеджерам — модель Гульшата)
- **Границы (callout):** НЕ управляет (Директор), НЕ настраивает KPI/тоглы/права, P&L закрыт, контент/ДЗ/солары у куратора, педраспределение у ст.куратора.

## 5. Вертикаль и system-graph (риск!)

Гард-тест `system-graph.test.ts` требует строгую ±1 вертикаль. Сплит вставляет Директора между Франчайзи и школьным персоналом (meeting §2):

```
Франчайзи-партнёр → Директор школы → { Бухгалтер · РОП(→Лидоруб/Менеджер/Маркетолог) · Старший куратор → Куратор }
                                       Администратор — сервис под Директором (приём/распределение)
```

- **Было:** `franchise ↔ school-admin ↔ senior-curator`.
- **Стало:** `franchise ↔ director ↔ senior-curator`; `director ↔ school-admin` (Админ — сервис); операционные связи школьного персонала ↔ `school-admin` сохраняются.
- **Действие:** обновить `system-map.ts` (иерархия + передачи), перецепить вертикальные crossLinks на `director`, синхронизировать гард-тест.
