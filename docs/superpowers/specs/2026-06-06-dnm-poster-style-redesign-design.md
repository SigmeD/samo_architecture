# Дизайн-спека: редизайн пользовательских экранов атласа в постер-стилистику

> Статус: **на согласовании** · Дата: 2026-06-06 · Автор: Max + Claude (web-designer mode)
> Класс документа: **намерение** (не зеркало кода). Источник правды по фактам ДНМ — **канон `samo-docs`** через MCP `project-docs`.
> Контекст: первый срез «Куратор» собран (L0→L1→L2 работают), но отрендеренные экраны «неинформативны» — плоская стена одинаковых карточек, ядро-процесс не читается, статусы и связи теряются. Эта спека описывает **визуальный редизайн** существующих экранов, не новую функциональность.

## 1. Цель и не-цель

**Цель.** Привести пользовательские экраны атласа (L0 обзор, L1 кабинет, L2 модуль) к полированной «постер-стилистике» эталонных HTML-постеров — с сильной иерархией, читаемым процесс-потоком, видимыми бейджами статусов и связями. Контент при этом — **строго по актуальному канону** (не копировать устаревшее из HTML-постеров).

**Не-цель.** Не меняем функциональность, маршрутизацию, контент-как-данные архитектуру, статический экспорт. Не трогаем исходные `docs/architecture/*.html` (остаются эталоном-первоисточником стилистики).

## 2. Источник стилистики (design language)

Эталон визуального языка — **`docs/architecture/dnm-architecture-poster.html`** (новый, полированный постер). Его система переносится в `styles/brand.css` (`@theme`-токены) + дженерик-компоненты `components/atlas/*`:

- **Типографика:** Fira Sans (заголовки/числа/eyebrow) + Roboto (тело) — уже подключены через `next/font`.
- **Каркас:** белый `frame` со скруглением 24–28px, мягкая тень, **верхняя градиент-полоса** (blue→blue-2→teal), фон страницы — двойной radial-glow + linear-gradient.
- **Шапка:** dot-texture, `eyebrow` (uppercase, letter-spacing, оранжевая засечка), крупный H1 (Fira, blue-d), строка-lead, **meta-чипы** (счётчики).
- **Секции:** нумерованный бейдж `01..NN` (синий скруглённый квадрат) + H2 + caption + затухающая линия.
- **Core-band:** синий градиент с feature-чипами / шагами процесса.
- **Карточки:** белые, левая 5px цветная полоса, икон-тайл, Fira-имя, буллеты с квадратными маркерами, бейджи.
- **Process-flow:** цепочка `step` со стрелкой ➤; варианты `alt` (оранжевый) / `end` (зелёный) / `branch`.
- **Result-band:** градиентный итоговый баннер.

**`dnm-cabinet-curator.html`** используется только как источник **структуры** кабинета (меню-плитки → ядро-лента → секции-карточки → связи → цикл + легенда), но НЕ его визуала (он старее) и НЕ его контента (содержит канон-нарушения, см. §5).

## 3. Жёсткие правила переноса (из CLAUDE.md, не нарушать)

- ❌ **Hex-литералов в компонентах нет** — только токены из `styles/brand.css` (`@theme`). Все цвета постера (`--blue`, `--green-l`, …) становятся бренд/зональными токенами.
- ❌ **Хардкода шрифтов нет** — `font-display` (Fira) / `font-sans` (Roboto) через CSS-vars.
- **Зона Куратора = СИНЯЯ** (D2). Старый HTML-кабинет зелёный — при порте перекрашиваем. Зелёный/оранжевый — только акценты (бейджи статусов), не фоны секций.
- **Статус не только цветом** — иконка/подпись (`done`/`partial`/`divergent`) рядом с цветом (a11y).
- Нативные `<a>`, крошки, focus-visible, контраст WCAG AA, адаптив (desktop ≥1280 / tablet 768–1279 / mobile <768).
- Каждый факт-норматив несёт `sources[]` (canon ID + версия).

## 4. Экраны

### 4.1 L0 — обзорный постер (`app/page.tsx`) — **ПЕРВЫЙ ЭТАП (старт)**
**Структурный порт `dnm-architecture-poster.html` 1:1**, контент — обновлён по канону. Требует **расширения контент-модели** `content/overview.ts` (сейчас урезана до hero+funnel+tiers). Делается **поэтапно по этажам** (см. §7а): каркас+дизайн-система → Потребители → Команда школы → Сеть → процессы/модули/result. Кабинеты, до которых этап ещё не дошёл, рендерят текущую краткую карточку-стаб (экран не ломается). Новая `OverviewSpec` несёт:
- `header.meta[]` — счётчики (кабинеты / модули / модели данных / семестры / языки / дата среза);
- `core` — ядро-движок: учебный движок · аналитика/KPI · геймификация (feature-чипы);
- `tiers[].cabinets[]` — богатые карточки кабинетов (имя, роль, иконка, цвет-зона, буллеты-фичи, бейджи `new`/`tag`/`opt`); ученик — hero (шире, 2-колоночные фичи);
- `processes[]` — ключевые сквозные процессы (воронка продаж, жизненный цикл договора) как `flow` со `step`/`alt`/`end`/`branch`;
- `crossModules[]` + `dataLayer[]` — сквозные модули (чипы) и слой данных/интеграций;
- `result` — итоговый баннер.

**⚠ Канон-верификация (обязательна перед шиппингом, §5).** Все числовые/нормативные факты постера (11 кабинетов, 73 модуля, 108 моделей, роялти 20%, тест ≥70%, лимиты наполнения 250/120, коэф. 70–80% и пр.) — НЕ копировать из HTML, а подтвердить через `get_doc`. Где канон расходится — править контент под канон, спорное помечать бейджем «на подтверждении».

### 4.2 L1 — кабинет (`app/cabinet/[slug]/page.tsx`)
**Синтез-макет утверждён** (моки `cabinet-poster-style.html`): шапка-герой (роль, lead, meta-сводка статусов) → `01 Ядро · процедура урока` (3 фазы, синий core-band, «на подтверждении») → `02 Домены` (4 тематические карточки в синей гамме, бейдж `implStatus` на каждом домене) → `03 Связи` (8 потоков с направлением ↔/←/→) → `04 Цикл проверки` + легенда статусов → result-баннер. Контент — из `content/cabinets/curator.ts` (уже канон-выверен). Группировка 15 доменов по 4 карточкам — **презентационная** (в каноне плоский список), фиксируется в коде.

### 4.3 L2 — модуль (`app/cabinet/[slug]/module/[moduleSlug]/page.tsx`)
Тот же визуальный язык: шапка-герой модуля + бейдж статуса, `ProcessFlow` в постер-стиле (step/alt/end), блок расхождений для `divergent` (`lesson-journal`, `homework-review`). Без отдельного нового контента — переверстка существующего.

## 5. Канон-верификация контента L0 (rule «НЕ ВЫДУМЫВАТЬ»)

Перед заполнением L0 — прогон фактов через MCP `project-docs` (`list_index` → `get_doc(<ID>)` со сверкой версии). Делегируется субагенту `general-purpose` на этапе плана. Категории к проверке:
- **Счётчики** шапки (число кабинетов / модулей / моделей данных / семестров / языков).
- **Ядро-движок** — формулировки геймификации (солары, не «оценки»), порог теста, RBAC-scope.
- **Карточки 11 кабинетов** — фич-буллеты каждого (особенно денежные: роялти, выплаты «Фикса+Бонус−Штраф»; лимиты групп; пробные уроки; «принято/не принято» без баллов).
- **Процессы** — воронка продаж, жизненный цикл договора (статусы без «заморозки»).
- **Сквозные модули и слой данных** — список сервисов платформы SAMO.

Любой неподтверждённый факт → бейдж «на подтверждении» (как в кабинете), не выдавать за канон.

## 6. Стратегия golden-screenshot (gate 2 vision-first-ui)

Старые PNG-постеры (`dnm-architecture-poster.png`, `dnm-cabinet-curator.png`) **перестают быть pixel-эталоном**: контент меняется под канон (L0) и кабинет уходит от зелёного оригинала (L1). Поэтому:
- Эталоны pixel-diff = **новые самоснимки** атласа (Playwright), коммитятся как baseline после визуальной приёмки.
- Исходные HTML/PNG остаются **референсом стилистики** (gate 1: «прочитать до кода»), не gate-2-целью.
- Это смена режима с «A: постер=North Star pixel» на **«канон=North Star данных, постер=North Star языка»** (решение владельца C/B, 2026-06-06).

## 7а. Поэтапность L0 (решение владельца 2026-06-06)

Старт — **основной постер L0**, поэтапно по этажам кабинетов. Каждый этап самодостаточен, экран остаётся целым (не дошли — стаб):
1. Каркас + дизайн-система (токены + компоненты постер-стиля) — один раз.
2. Этаж **«Потребители»** (ученик-hero, родитель, гость) + канон-сверка.
3. Этаж **«Команда школы»** (куратор, ст. куратор, админ, менеджер, финансист).
4. Этаж **«Сеть и управление»** (франчайзи, куратор франшиз, руководитель).
5. **Ядро-движок** + **ключевые процессы** (воронка + жизненный цикл договора) + **сквозные модули и платформа** + **данные и интеграции** + **result**.
6. Golden-screenshot L0 + обновить STATUS. После L0 — переход к L1 (кабинет), затем L2.

## 8. Открытые вопросы

- **Группировка 4 доменных карточек L1** — закрепляем «Проверка и оценивание / Уроки и занятия / Отчётность / Развитие и связь» (по образцу cabinet-poster). Альтернатива — по `implStatus`. **Решение: по смыслу** (утв.).
- **Цвет доменных карточек L1** — монохром-синяя гамма (blue / blue-2 / teal / slate), чтобы не заимствовать семантику чужих зон (зелёный=ученик, оранжевый=админ). Цвет несут только бейджи статусов. **Решение: монохром-синий** (по умолчанию; пересмотреть, если владелец захочет богаче).
- **Объём L0-карточек кабинетов** — 11 полных карточек с фичами (как постер) против кратких. **По умолчанию: полные**, контент по канону.
