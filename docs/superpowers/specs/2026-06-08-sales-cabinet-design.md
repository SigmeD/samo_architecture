## 2. Ядро = «Воронка лида-родителя» (до передачи; БЕЗ оплат)

Per spec §5 — менеджер ведёт **только** воронку лида до согласия, дальше передаёт администратору:

1. 🎯 Лид (родитель) — приём от маркетолога, **speed-to-lead** (контакт в минуты), приоритизация
2. 📋 Квалификация (CRM, анкета) → рекомендация продукта + скрипты
3. 🎬 Демо-доступ + запись на пробное (⚠ **пробное проводит КУРАТОР**, не менеджер)
4. 💬 Дожим — омниканально (WhatsApp Business API ↔ CRM), отработка возражений
5. 🤝 Согласие → **передача администратору**
6. ✅ Передано на оформление (договор+оплата у администратора; менеджер видит статус read)

**⚠️ ПОПРАВКА К ПОСТЕРУ + canon-proposal:** постер (02.06) показывает менеджера, который проводит пробное, оформляет **договор, оплату, рассрочку, наличную, дебиторку, возвраты**. Spec §17/§19 это **опровергает**: пробное — куратор; договор/оплата/рассрочка/возвраты/расторжение — администратор/финансы; оплату подтверждает бухгалтер; менеджер только закрывает продажу и видит статус (read). ⚠ расходится с `CONV-RBAC-DNM-001` v1.4 (Мен имеет `CRU` приём оплаты + рассрочка/график, `R` дебиторку; сводка канона даёт Мен «проведение пробного урока») → **canon-proposal**.

## 4. Связи (направленные)

- **Маркетолог** (`⇄`) — передаёт лиды-родители (источник: таргет/органика/реклама/рекомендация); менеджер даёт обратную связь по качеству лидов и причинам отказа
- **Куратор** (`⇄`) — менеджер выдаёт демо-доступ и записывает → **куратор проводит пробное**; коэф. успешности 70–80%
- **Администратор школы** (`→`) — передаёт согласившегося клиента → **админ оформляет договор + оплату + зачисление** (мастер)
- **Финансист/бухгалтер** (`→`) — бухгалтер подтверждает оплату; дебиторка/возвраты у админа/финансов; менеджер видит статус (read)
- **Куратор франшиз** (`←`) — обучающие созвоны для менеджеров (Академия)
- **Руководитель проекта** (`→`) — KPI и воронка продаж → головной офис

## 7. Расхождения с каноном → `docs/canon-proposals/2026-06-08-sales-cabinet.md`

1. **Пробное занятие проводит КУРАТОР, не менеджер.** Канон-сводка RBAC v1.4: «Менеджер … проведение пробного урока». Spec §17.1 + встреча 04.06: демо-доступ+запись — менеджер, **пробное проводит куратор**.
2. **Менеджер не ведёт оплату/рассрочку.** Канон RBAC v1.4 §6: Мен «Приём оплаты/отметка офлайн» = `CRU`, «Рассрочка/график» = `CRU`. Spec §10/§19: договор/оплата/рассрочка → администратор/финансы; бухгалтер подтверждает.
3. **Дебиторка — read-only-статус, не управление.** Канон §6: Мен «Дебиторка» = `R`. Spec: у менеджера только просмотр статуса по своим клиентам.
4. **Возвраты — не у менеджера** (канон уже даёт Мен «—», постер ошибочно показывает «Возвраты»).
5. **Cross-ref системной карты:** строка «Менеджер по продажам» в `/map` отражает канон (`CRU` платежи) — при принятии proposal флипнуть финансовые ячейки Мен с ⚠.
