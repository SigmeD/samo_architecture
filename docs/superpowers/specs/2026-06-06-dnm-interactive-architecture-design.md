## 1. Контекст и проблема

В `docs/architecture/` лежат ранее построенные верхнеуровневые постеры архитектуры ДНМ — обзорный (`dnm-architecture-poster*.png/html`) и по 11 кабинетам (`dnm-cabinet-*.png/html`), плюс `frontend-architecture.md`, 5 ADR и брендбук. Постеры **статические** (PNG + печатный HTML на inline-CSS) и местами **устарели** (использовались неактуальные данные).

**Задача:** сделать архитектуру **интерактивной** и **актуальной**. Обзорный постер — главная страница; клик по кабинету ведёт на страницу кабинета; в будущем клик по модулю — на бизнес-процесс модуля. Архитектура описывается как **бизнес-процессы**: согласуется с заказчиком и передаётся разработчикам. Аудитория — и разработчики, и владелец бизнеса. ДНМ — детская образовательная система предпринимательства (10–17 лет); ключевая бизнес-цель клиента — **продажа франшизы**, что влияет на архитектуру.

## 2. Цели и не-цели

**Цели:**
- Навигируемый «атлас»: постер → кабинет → модуль (drilldown в бизнес-процесс).
- Контент **актуален и привязан к канону** (трассировка: ID + версия документа).
- Видимый статус реализации (badge `done/partial/planned/divergent`) — «намерение vs код».
- Переиспользуемые компоненты **внутри проекта**; деплой на Vercel.
- Пиксельная верность исходным постерам (их показывают владельцу).

**Не-цели (YAGNI):**
- Не функциональный продукт: нет бэка, нет реальных мутаций, нет авторизации, нет MSW/Zustand. Это **презентационный** атлас на типизированном контенте.
- Не переиспользуем код/компоненты `samo-dnm` (только как информационную базу знаний).
- Не строим все 11 кабинетов сразу — сначала вертикальный срез (Куратор).

## 6. Модель контента (типы)

Единый источник `content/types.ts`, зеркалит структуру верифицированных данных 1:1:

```ts
type ImplStatus = 'done' | 'partial' | 'planned' | 'divergent';
type Direction  = 'in' | 'out' | 'both';

interface RoleMeta       { code: string; title: string; emoji: string }
interface SourceCitation { id: string; version: string; section?: string }
interface ProcessStep    { n: number; title: string; desc: string; actors: string[]; source: string; gamification?: string }
interface ProcessFlow    { title: string; badge?: string; steps: ProcessStep[]; note?: string }
interface DomainSpec     { title: string; items: string[]; source: string; readOnly?: boolean; toggleable?: boolean }
interface CrossLink      { toCabinet: string; label: string; direction: Direction; source: string }
interface ModuleRef      { slug: string; title: string; status: ImplStatus; summary: string }
interface CabinetSpec {
  slug: string; role: RoleMeta; zoneColor: string; purpose: string;
  coreProcess: ProcessFlow; sectionTiles?: {icon:string;title:string;isHub?:boolean}[];
  domains: DomainSpec[]; crossLinks: CrossLink[]; modules: ModuleRef[];
  sources: SourceCitation[]; implStatus: ImplStatus;
}
interface ModuleSpec {  // страница drilldown
  slug: string; cabinetSlug: string; title: string; status: ImplStatus; summary: string;
  purpose?: string; process?: ProcessFlow; domains?: DomainSpec[];
  crossLinks?: CrossLink[]; sources: SourceCitation[];
}
```
**Правило:** свободнотекстовые `source` на шагах/доменах оставляем как есть (не парсить — иначе «выдумаем» номера разделов); структурный `sources[]` уровня кабинета/модуля питает компонент `SourceRef`.

## 7. UX-модель интерактива

- **3 уровня:** L0 постер-обзор → L1 кабинет → L2 домен/модуль. На каждом узле известен путь вверх (хлебные крошки + «← Назад к постеру»).
- **4 состояния блока** (единый протокол): `default` (зональный акцент) · `hover/focus-visible` (подъём, тень, иконка ↗; клавиатура = мышь) · `active/current` (`aria-current`, бейдж «вы здесь») · `«в работе»` (planned/divergent или ещё не сделано: пониженная насыщенность + штриховка + бейдж + `aria-disabled`, клик = tooltip, **не** пустой серый прямоугольник).
- **Брендбук:** SAMO Blue — каркас/хром (шапка, крошки, кнопки, фокус-кольцо). Оранжевый/зелёный — **только акценты**, не фоновые секции. Шрифты: Roboto (UI/тело) + Fira Sans (заголовки/числа). Логотип из `public/brand/logo.svg`.
- **Зональная палитра** (семантика поверх бренда, в `content/zones.ts`): синий=управление, зелёный=учёба/ученик, оранжевый=школа/админ, фиолетовый=франшиза, бирюзовый=продажи/контроль, золотой=финансы. **Куратор = синий** (D2). Зона красит только акцент-полосу/бордеры/бейджи.
- **Подсветка связей:** чип роли-адресата красится в **зональный цвет адресата** и является ссылкой; направление — стрелкой (→ исходящий, ← управляющее сверху, ⇄ both). Hover синхронно подсвечивает связанную карточку.
- **Адаптив:** убрать фикс-ширину постеров для экрана (оставить для `@media print`); брейкпоинты desktop ≥1280 / tablet 768–1279 (2 кол.) / mobile <768 (1 кол., funnel → вертикальный timeline).
- **Доступность:** нативные `<a>`; видимый `:focus-visible` (синий); контраст — для текста <18px брать тёмные варианты акцентов (`--orange-d`, `--green-d`); статус не только цветом (иконка+текст); `prefers-reduced-motion`; landmark-разметка.
- **Badge `implStatus`:** `done`→зелёный ✓ · `partial`→оранжевый ⚙ «частично» · `planned`→нейтрально-синий ⏳ «план» (**не** красный) · `divergent`→предупреждающий ⚠ «расхождение с каноном» + tooltip с пояснением. На постере у карточки — сводный наихудший статус.

## 8. Кабинет Куратора — верифицированные факты (канон)

**Роль:** `op-kurator-dnm` · «Куратор ДНМ» · зона **синий** (D2; данные BA давали зелёный — переопределено) · `implStatus: partial`.

**Назначение:** операционный кабинет педагога-куратора. Куратор — **не оценщик, а модератор/наставник**: проводит уроки во встроенном видеосозвоне по процедуре урока, проверяет ДЗ/классную работу (**принято / на доработку, без баллов**), даёт безоценочную обратную связь, **вручную начисляет солары** за субъективные работы, подтверждает посещаемость, формирует отчёты родителям, видит свой KPI/статистику, проходит обучение/тесты куратора. Работает строго в scope своих групп своей школы (`schoolId+groupId`). **Финансы/оплаты в кабинете не отображаются.**
*Источники: SPEC-M3-DNM-001 v2.0 §2.1; SPEC-DNM-FUNC-001 v2.1 §3; REG-DNM-CURATOR-001 v1.0; CONV-RBAC-DNM-001 v1.2.*

**Ядро-процесс — «Процедура проведения онлайн-урока» (3 фазы)** *(заменяет отменённый чек-лист «7 этапов»)*:
1. **Приветствие** — войти за 5–10 мин, проверить технику/материалы, поздороваться, назвать тему, создать атмосферу. *(REG-DNM-LESSON-001 v1.0 §1)*
2. **Основная часть** — программа и тайминг, примеры/интерактив, вовлечение, чат, дисциплина; внутри — проверка ДЗ/классной, безоценочная ОС (техника «Бутерброд»: позитив → зона роста → поддержка), ручное начисление соларов. *(REG-DNM-LESSON-001 §2; REG-DNM-CURATOR-001; SPEC-DNM-TZ-001 v3.2 §4.4)*
3. **Завершение** — итоги, ОС по активности, ответы на вопросы, выдача ДЗ, прощание; подтверждение списка посещаемости (онлайн засчитывается при участии ≥40 мин). *(REG-DNM-LESSON-001 §3; SPEC-M3-DNM-001 v2.0 §3.6)*

> ⚠️ **На подтверждении (open question):** замена «7 этапов» — это **намерение**, финальный пакет процедур передаёт руководитель франшизы (br7). В каноне **два кандидата** структуры: 3-фазная (`REG-DNM-LESSON-001`) и 9-этапная методичка (`REG-DNM-CURATOR-001`: Приветствие → Проверка ДЗ → Пояснение ДЗ → Перерыв → Практика → Рефлексия → Фиксация → Оставшееся время → Завершение). В атласе показываем 3 фазы как ядро + 9-этапную как вложенную методичку, обе с пометкой «на подтверждении». Модуль `lesson-journal` = `divergent`.

**Домены (15):** Мои группы · Журнал занятий (процедура урока ⚙) · Проверка ДЗ · Классные/эссе/бизнес-проекты (3 цикла) · Посещаемость · Успеваемость группы · Отчёты родителям · Геймификация (начисление соларов) · Расписание · Видеосозвоны · Все уроки / доступ к программе 🔒 · Обучение куратора / «Академия» · Мой KPI 🔒 · Пробные уроки · Сертификаты/Документы/Задачи/Чат/Уведомления.
