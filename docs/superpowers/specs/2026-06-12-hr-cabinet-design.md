## 2. Ядро

«**Воронка найма → онбординг → аттестация → допуск → удержание**» (badge «кадровый контур»), 6 шагов:
1. 🔎 Источник → скрининг (обзвон, шаблон вакансии)
2. 📝 Тест при приёме (банк тестов по роли → балл → допуск)
3. 🗣️ Собеседование → оффер
4. 🎓 Онбординг-Академия (видео+текст по модулю)
5. ✅ Аттестация → допуск к работе с клиентами (баллы → пробный период)
6. 📉 Удержание (мониторинг текучести, exit-интервью)

## 3. Домены (8) — из `REG-DNM-HR-001 v1.1 §2–§3`

1. **🎯 Воронка рекрутинга** — стадии + метрики (обзвонил/пригласил/дошёл до теста/нанято), шаблоны вакансий, кадровый резерв по ролям (МОП/РОП/куратор/ст.куратор/админ/маркетолог/таргетолог). _ATS-автоматизация — ⏸ planned, conditional ADR-0002._ `source: REG-DNM-HR-001 v1.1 §2.1`
2. **📝 Банк тестов при приёме** — HR загружает тесты по ролям; кандидат проходит → балл → допуск к собеседованию/пробному. `§2.2`
3. **🎓 Академия онбординга / аттестация** — role-based обучение (видео+текст обязанностей → тест после модуля → баллы = аттестация → пробный → допуск). Платформа = **M6-LMS**; контур — внутри HR (решение владельца 12.06, `OQ-HR-01` закрыт). `§2.2`
4. **📊 KPI-карты персонала** — HR проектирует/ведёт KPI-карты ролей (метрики, веса, пороги). **Граница (`OQ-HR-02`):** HR задаёт *что* мерить + таргеты; движок `SPEC-KPI-PAYOUT` считает *выплату* (Фикс+Бонус−Штраф). KPI также питают репутационные рейтинги (`SPEC-DNM-RATING §4`). `§2.3`
5. **🗂️ База данных персонала** — system of record: карточка сотрудника (роль, школа, дата приёма, статус, документы, **ставка как кадровый атрибут — не платёж**, история ролей/переводов, результаты тестов/аттестаций, табель/отпуска). Питает ФОТ (`SPEC-KPI-PAYOUT §2.4`), оргструктуру, IAM/M1 (создание/деактивация учёток). ПДн — `OQ-HR-03`. `§2.4`
6. **🤝 Внутренняя политика / дисциплина / медиация** — регламенты входа/выхода, этикет, контроль; медиация конфликтов, дисциплинарные процедуры (`REG-ETHICS-001`). `§2.5`
7. **📉 Удержание (антитекучка)** — мониторинг текучести по ролям/школам, exit-интервью, программы удержания. `§2.6`
8. **📈 KPI самого HR** — воронка-конверсия, time-to-hire, качество найма (retention 3/6/12 мес), текучесть↓, укомплектованность. Таргеты — `OQ-HR-04`. `§3`

## 4. Связи

- **`director`** (both) — вертикаль: HR подчиняется директору школы; согласование KPI-карт, оргструктура, отчёты по найму/текучести вверх. _Director получает реципрокную связь._
- **`finance`** (out) — кадровый handoff: KPI-карты + база (ставки) → бухгалтер для ФОТ/расчёта выплат (HR не считает выплаты — граница `OQ-HR-02`). Одностороннее (HR→finance), реципрокность не требуется.

> Инферред-связи (НЕ заявлены каноном — не добавляем): ст.куратор/куратор франшиз (наём кураторов), франчайзи (сетевая кадровая политика). Канон называет их как *роли, на которые HR нанимает*, не как контрагентов.

## 5. Границы (callout)

HR **НЕ**: видит учебные данные (≠ куратор/ст.куратор); видит финотчёты/P&L/роялти (≠ бухгалтер/владелец); считает выплаты (это движок KPI-PAYOUT + бухгалтер); распределяет учеников (это админ — HR распределяет *персонал*). RBAC per-resource — отложен каноном в `OQ-ORG-03` (в `/map` показываем scope, не CRUD-ячейки).
