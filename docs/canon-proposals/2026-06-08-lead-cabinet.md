Новое клиентское намерение по кабинету руководителя (франчайзера) расходится с текущим каноном по **двум сильным**
пунктам (авторство учебной программы; границы видимости данных ученика) и **добавляет** набор новых подсистем

## L1 — `CONV-RBAC-DNM-001`: авторинг учебной программы (руководитель → куратор франшиз)

- **id:** `CONV-RBAC-DNM-001` · **версия:** **1.4** · **тип:** CHANGE (прямое расхождение нового намерения с действующим RBAC)
- **Канон сейчас (`docs/structure/dnm/dostupy-roley-dnm.md`):**
  - матрица: `| Конструктор программы (уроки/ДЗ/тесты) | … | CRUD |` — **CRUD только у `br7`** (руководитель), у всех прочих «—», включая куратора франшиз;
  - текст: «**Руководитель франшиз / проекта (br7):** … **конструктор программы (отражается во всех школах)** …».
  - То есть **старый постер (срез 02.06) «только руководитель авторит программу» был канон-корректен.**
- **Новое намерение (проектный док §4/§15; follow-up §3):** авторинг/редактор/публикацию учебной программы ведёт
  **КУРАТОР ФРАНШИЗ** (как держатель настройки программы; «методист/контент» — право внутри его роли);
  руководитель **ПОДТВЕРЖДАЕТ** программу (gate качества) + стратегический контроль. Публикация = дифф/предпросмотр,
  раскатка пилот→сеть, откат, версии (follow-up §1).
- **Предложение:** перенести `CRUD` строки «Конструктор программы» с `br7` на роль куратора франшиз
  (`op-kurator-franshiz-dnm`); ввести у `br7` право **Approve/Publish-gate** учебной программы (подтверждение версии).
  Согласовать с `SPEC-DNM-TZ-001` v3.8 (раздел кабинета руководителя/куратора франшиз) и `SPEC-DNM-FUNC-001` v2.3.
- **rationale:** на 06–08.06 владелец сместил центр тяжести кабинета руководителя с «зауча/редактора программы» на
  стратегию/финансы сети («здесь не люди — здесь цифры», встреча 05.06), а операционный авторинг программы закрепил
  за куратором франшиз. Это изменение ролевой модели, не внутреннее противоречие — **подаём только после «да».**

## L2 — `CONV-RBAC-DNM-001` / `SPEC-DNM-RATING-001`: руководитель не видит данные отдельного ученика

- **id:** `CONV-RBAC-DNM-001` v1.4, `SPEC-DNM-RATING-001` v1.2 · **тип:** CLARIFY / CONSTRAIN
- **Канон сейчас:** `br7` имеет широкий `R(все)` (в т.ч. «Программа курса ребёнка R(все)», аналитика), без явного
  запрета проваливаться в отдельного ученика.
- **Намерение (проектный док §9/§16):** на уровне руководителя — **только сводные цифры по франчайзи**; прогресс и
  сертификаты конкретного ученика **не видны** (это уровень куратора/администратора/родителя). Зона риска RP7 —
  агрегат «без имён».
- **Предложение:** зафиксировать ограничение «`br7` — агрегат по франчайзи/школам, без PII-уровня ученика» как
  явное RBAC-правило (граница приватности), уточнив `R(все)` до `R(агрегат)` для ученических ресурсов.

## L3 — `SPEC-M10-FRANCHISING-001`: целостность роялти на наблюдаемом участии + видеофиксация

- **id:** `SPEC-M10-FRANCHISING-001` v1.1 (+ `ARCH-SERVICEMAP-001` v1.1) · **тип:** ADD
- **Намерение (follow-up §1/§9):** основной контроль занижения числа детей → **наблюдаемое платформой участие**
  (кто реально подключился к онлайн-уроку) + **видеофиксация** офлайна; LMS-vs-финотчёт — третий слой.
  «Активного» определяет система по участию, **не франчайзи**.
- **Предложение:** дополнить M10 (роялти/контроль) механизмом «целостность роялти = участие на платформе +
  видеофиксация (`ARCH-SERVICEMAP-001`)»; связать с замкнутым циклом корректирующих действий.
