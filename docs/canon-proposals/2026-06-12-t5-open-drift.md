## C1 — Trial-runner: пробный урок ведёт куратор vs менеджер
**Статус канона:** само-противоречие, НЕ ратифицировано.

- `SPEC-DNM-REG-001 v1.4 §3`: пробное ведёт **менеджер** (гибко — допустим куратор).
- «Гибкость пробного» (две модели) — `SPEC-M3-DNM-001 v2.10` Волна 2, **не ратифицирована**.

**Атлас сейчас:** sales/guest/curator подают «менеджер ведёт пробное сам либо назначает куратора» как решённый реверс.
**Предложение:** понизить до «на согласовании» во всех трёх кабинетах (не утверждать как факт), пока владелец не разрешит TZ §13.4 ↔ REG §3. Затрагивает: `sales.ts`, `guest.ts`, `curator.ts`.

## C2 — Marketer: close-rate / выручка vs «ноль финансов» (тяжёлый) — ✅ РАЗРЕШЁН 12.06
**Решение владельца (12.06):** «маркетолог не видит выручку» → атлас выровнен под канон «ноль финансов» (commit `93eeea3`). Ядро marketer переписано на «Контур лидогенерации → квалифицированный лид»; выручка/close-rate/ROAS/CAC = уровень ГО (Маркетолог ГО), маркетолог франчайзи их НЕ видит; KPI = доля квалиф. лидов (R). `divergent`-флаги сняты, кабинет ALIGNED. `propose_change` не нужен.

**Статус канона:** ратифицировано ПРОТИВ атласа.
- `CONV-RBAC-DNM-001 v1.6 §9 «Маркетинговые ресурсы»` + решение «ноль финансов» (11.06): Мрк **«Close rate / выручка по источнику» = «—»** (только ГО). KPI Мрк = «доля квалифицированных лидов» (R).

**Атлас сейчас:** ядро `marketer.ts` = «Контур лидогенерации со сквозной атрибуцией», dashboard «close rate по источнику», модуль attribution-pipeline, crossLink finance «событие конверсии → close rate возвращается маркетологу». Это **противоречит** канону: маркетолог НЕ видит выручку/close rate.
**Предложение (нужно решение владельца):** либо (а) перенести видимость close-rate/выручки на уровень ГО (маркетолог видит только качество лидов/CPL, не оплаты), приведя ядро marketer к «ноль финансов»; либо (б) явный owner-divergent canon-proposal (атлас намеренно даёт маркетологу сквозную атрибуцию вопреки RBAC v1.6 §9) → `propose_change` в канон. До решения — пометить ядро/атрибуцию `divergent` + «на согласовании».

## C3 — Авторинг учебной программы (OQ-ORG-03) — ✅ ЗАКРЫТ 12.06
**Решение владельца (12.06):** «авторинг — подтверждаю» → **методист = автор** программы. Атлас (commit `28d5186`): снят «на согласовании»/OQ-ORG-03 с авторства в `lead.ts`+`methodist.ts`; цепочка **методист авторит → руководитель (br7) утверждает (gate, сам не правит)** — ратифицировано `SPEC-M3-DNM-001 v2.10 FR-M3-013/101` + подтверждено владельцем. КФ — организационный руководитель методиста (ревью), финальный gate — руководитель. RBAC v1.6 §2 (CRUD у br7) — старее, перекрыт M3 v2.10 + решением.

**Статус канона:** три конкурирующие модели, не ратифицировано (`OQ-ORG-03` открыт).
- `CONV-RBAC-DNM-001 v1.6 §2`: «Конструктор программы» = CRUD у **br7** (руководитель), остальные «—».

**Предложение:** перенацелить предложенного автора в атласе с «куратор франшиз» → **методист** (совпадает с M3 v2.10 + решением встречи); граница методист↔программа = `OQ-ORG-03`, оставить «на согласовании». `methodist.ts` уже близок (см. корзина B: divergent→ратифицировано для самого механизма «методист готовит, br7 утверждает»; спор только про роль КФ-ревью).

## C5 — Senior-curator: линия отчётности
**Статус канона:** многослойно.
- Орг-подчинение (дерево Screenshot_4, `ROLES v1.6 §2б`): Ст.куратор под **Директором школы**.
- In-school эскалация качества (`REG-DNM-SCHOOL-ADMIN v1.2 §6.1`): куратор → ст.куратор → **администратор**.
- ГО-данные (`OQ-ORG-04`, `HIERARCHY v1.11`): школьные ст.кураторы → **Куратор франшиз** (заменяет понижённого «Ст.куратора ГО»).

**Атлас сейчас (после T3):** ст.куратор ↔ director (орг-вертикаль) + ↔ school-admin (операционка). Это покрывает орг-подчинение + операционную эскалацию, но лейблы домена «отчёты → директору» не отражают: качество → админ, ГО-данные → КФ.
**Предложение:** уточнить лейблы (качество → админ; ГО-агрегат → КФ), орг-подчинение директору оставить. Нюанс, не реверс. Можно применить как безопасный лейбл-фикс ПОСЛЕ мержа PR #10 (чтобы не плодить конфликты с уже изменёнными в T3 связями).
